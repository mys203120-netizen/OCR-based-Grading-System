# PRD: CTD 영어 서술형 OCR 채점 시스템

## 1. 제품 목적

조교가 폰스캔한 영어 서술형 답안지 PDF를 업로드하면, 시스템이 학생 1명 단위로 OCR과 1차 채점을 수행하고 강사/조교가 검수 후 학생 및 학부모에게 개인화 피드백을 발송할 수 있게 한다.

핵심 가치는 채점 시간 단축, 원문 보존, 강사별 데이터 분리, 학생 성장 데이터 축적이다.

## 2. 사용자

- 강사: 본인 수업 학생의 답안, 랭킹, 성장세만 조회한다.
- 조교: 학생 등록, PDF 업로드, 동명이인 확정, 채점 검수, 문자 발송을 수행한다.
- 관리자: 강사/수업 생성, 전체 운영 설정, 문자 발송 provider 설정을 관리한다.

## 3. 필수 기능

### 3.1 강사/수업 권한

- 모든 학생, 시험, 답안지, 대시보드는 수업에 귀속된다.
- 수업은 강사에게 귀속된다.
- API는 `X-Instructor-Id` 기준으로 현재 강사의 수업 데이터만 반환한다.
- 다른 강사의 수업, 학생, 답안지 ID를 직접 호출해도 404로 처리한다.

### 3.2 학생 등록/수정/삭제

- 조교는 학생을 등록할 수 있다.
- 필수: 이름, 소속 수업.
- 선택: 학부모 전화번호, 학생 전화번호, 등급대, 학년, 학교, 메모, 외부 학생 ID.
- 학생 정보는 수정 가능해야 한다.
- 학생 삭제는 soft delete로 처리한다.
- 삭제된 학생은 명단과 자동 매칭 대상에서 제외한다.
- 과거 시험 기록, 답안지 이미지, 발송 기록은 보존한다.

### 3.3 PDF 업로드 및 페이지 분리

- 조교가 업로드할 때 입력하는 값은 네 가지뿐이다.
- 입력값 1: 반 선택.
- 입력값 2: PDF.
- 입력값 3: 서술형 문항 총 개수.
- 입력값 4: 문항별 강사 코멘트.
- 시험명은 시스템이 자동 생성한다.
- 시험 날짜는 업로드 날짜로 자동 저장한다.
- OCR 채점 후 시험 날짜 수정은 가능해야 한다.
- PDF는 페이지별 PNG 이미지로 저장한다.
- 각 페이지는 독립 submission으로 처리한다.
- 한 번에 PDF 전체를 모델에 보내지 않는다.
- 5문항 시험처럼 문항 수가 10개가 아닌 시험도 처리한다.
- 학생이 총 문항 수보다 적게 작성한 경우 미작성 문항은 자동 오답, 0점으로 반영한다.

### 3.3.1 반 및 학년 자동 관리

- 강사마다 여러 반을 가질 수 있다.
- 강사 생성 시 기본 반으로 `전체 학생`, `예비고1`, `고1`, `고2`, `고3/N수`를 자동 생성한다.
- 학생 등록 시 등록 기준 학년과 등록 기준 학년도를 저장한다.
- 학년도는 3월 시작 기준이다.
- 시간이 지나면 현재 학년은 자동 계산된다.
- 예: 2026학년도에 고1로 등록한 학생은 2027년 3월부터 고2로 표시된다.
- `전체 학생` 반은 해당 강사의 활성 학생 전체를 포함한다.
- `고1`, `고2`, `고3/N수` 반은 현재 자동 계산된 학년 기준으로 학생을 포함한다.
- 학생 등록 시 별도 운영 반이 없으면 수동 반 선택은 생략할 수 있다.
- 별도 운영 반이 있으면 학생 등록/수정 시 해당 수동 반만 선택한다.

### 3.4 OCR

- OCR 모델은 학생 필기 원문만 JSON으로 복원한다.
- 문법, 철자, 어순, 구두점을 수정하지 않는다.
- 화살표, `^`, `<...>` 삽입구는 물리적 위치에 맞게 문장 안에 넣는다.
- 빨간 펜 채점 흔적은 제외한다.
- 읽을 수 없는 글자는 `[Unrecognizable]`로 표시한다.

### 3.5 채점 및 피드백

- 채점 모델은 OCR JSON과 강사 코멘트/모범답안/루브릭만 사용한다.
- `raw_ocr_text`는 불변 데이터다.
- 개선 문장은 `better_sentence`에만 작성한다.
- 문항별 점수, 총점, A/B/C/D 등급을 만든다.
- 학생별로 부족한 개념, 단어, 문법, 어순, 의미 누락을 분류한다.
- 다음 시험 기대 여부는 누적 데이터 기반으로 생성한다.
- 학생에게 보이는 피드백은 AI 보고서 말투가 아니라 강사가 직접 써준 말투여야 한다.
- 문항별 피드백은 실제 오류, 왜 문제인지, 다음번 체크포인트를 포함한다.
- 금지 표현 예: "전반적으로", "학습자", "해당 문항", "개선이 필요합니다", "역량", "도움이 됩니다".
- 시스템은 AI식 표현이나 너무 짧고 일반적인 피드백을 감지해 검수 플래그를 남긴다.

### 3.6 동명이인 처리

- OCR 이름이 수업 명단에서 1명과 일치하면 자동 연결한다.
- 같은 이름이 여러 명이면 자동 연결하지 않는다.
- submission에 후보 학생 목록을 저장하고 `needs_human_review=true`로 표시한다.
- 사용자는 마지막 검수 단계에서 누구의 답안지인지 선택한다.
- 선택 후 `student_match_status=manual_matched`로 변경한다.

### 3.7 답안지 이미지 조회

- 학생 상세 화면에서 시험 이력을 볼 수 있어야 한다.
- 시험 날짜를 누르면 해당 시험의 답안지 이미지와 OCR/채점 결과를 볼 수 있어야 한다.
- 이미지 경로는 submission에 저장한다.

### 3.8 발송

- 채점 결과, 총점, 개인화 요약, 보강 방향을 문자 초안으로 생성한다.
- 수신자는 학부모 또는 학생을 선택할 수 있다.
- 전화번호가 없는 경우 발송 불가 상태를 반환한다.
- 발송 채널은 카카오 알림톡을 기본으로 한다.
- SOLAPI 알림톡 provider를 통해 실제 발송 요청을 생성한다.
- 발송 요청 접수 후 메시지 상태는 `queued`로 저장한다.
- provider 오류가 발생하면 `failed`와 오류 메시지를 저장한다.
- 로컬 QA에서는 `MESSAGE_PROVIDER=mock`으로 provider 없이 상태 흐름을 확인할 수 있다.

## 4. 대시보드

- 시험별 랭킹을 제공한다.
- 학생별 성적 변화 추이를 제공한다.
- 검수 필요 답안 수를 제공한다.
- 동명이인/미매칭/읽기 불가 OCR은 검수 대상으로 표시한다.

## 5. 비기능 요구사항

- Gemini OCR 모델과 채점 모델은 환경변수로 분리한다.
- 기본 모델은 `gemini-3.5-flash`다.
- 추후 `gemini-3.6-flash` 등으로 모델명만 교체 가능해야 한다.
- 데이터 원본성 보존을 위해 OCR JSON, 검수 수정 JSON, 채점 JSON, 검수 채점 JSON을 분리 저장한다.
- 운영 DB는 Postgres를 기본으로 한다.
- 운영 작업 큐는 Redis Queue(RQ)를 기본으로 하며 API 서버와 워커를 분리한다.
- 문자 provider는 SOLAPI 카카오 알림톡을 기본으로 한다.

## 6. 성공 기준

- 조교가 PDF 업로드 후 학생별 1차 채점 결과를 확인할 수 있다.
- 동명이인 답안지는 자동 연결되지 않고 후보 목록을 보여준다.
- 강사는 본인 수업 학생 데이터만 조회할 수 있다.
- 학생 상세에서 시험 날짜별 답안지 이미지를 열람할 수 있다.
- 문자 초안 생성과 발송 상태 저장이 가능하다.
